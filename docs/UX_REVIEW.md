# FigmaLab VSCode Extension — UX Review

**작성일**: 2026-03-07
**리뷰어**: UX 전문가 관점 분석
**대상 버전**: 0.1.1
**분석 범위**: 사용성(Usability), GUI, 사용자 경험(UX), 정보 구조(IA), 접근성(Accessibility)

---

## 요약

FigmaLab은 Figma MCP 연동과 AI 코드 생성이라는 명확한 가치를 제공하는 VSCode 확장이다. 그러나 현재 구현은 "기능 동작"에 집중되어 있고, 사용자가 별도의 지식 없이 이 확장을 처음 실행했을 때 어떻게 동작하는지에 대한 고려가 부족하다. 패널 순서, 언어 일관성, 액션 버튼 위치, 온보딩 흐름 등 여러 영역에서 즉각적인 개선이 필요하다.

---

## 1. 정보 구조(Information Architecture) 문제

### 1-1. 패널 순서가 실제 워크플로우와 불일치

**현재 패널 순서** (package.json 기준):
```
Agent → Figma → Prompt → Log
```

**실제 사용 워크플로우**:
```
Figma(연결) → Agent(설정) → Prompt(생성) → Log(확인)
```

사용자는 VSCode 사이드바에서 위에서 아래로 읽는다. 첫 번째로 보이는 "Agent" 패널은 실제로 두 번째 단계다. 처음 접하는 사용자는 어디서 시작해야 하는지 알 수 없다.

**권장 순서**:
```
Figma → Agent → Prompt → Log
```

---

### 1-2. 패널 간 의존성이 UI에 표시되지 않음

코드를 생성하려면 다음 세 가지가 모두 완료되어야 한다:
1. Figma 패널에서 MCP 데이터 fetch
2. Agent 패널에서 API 키 저장 + 모델 로드 + 저장
3. Prompt 패널에서 Generate 실행

그러나 UI에는 어떤 단계가 완료되었는지, 다음에 무엇을 해야 하는지 알려주는 시각적 단서가 전혀 없다. 사용자는 Prompt 패널에서 Generate를 눌렀을 때 실패하고 나서야 이전 단계가 빠졌음을 인지한다.

**권장**: 각 패널 헤더에 완료 상태 표시 (체크 아이콘 등), 또는 첫 실행 시 단계별 안내 흐름 제공.

---

## 2. 온보딩 경험 부재

### 2-1. 신규 사용자를 위한 가이드가 없음

사이드바를 처음 열면 세 개의 빈 패널이 나타난다. "어떻게 시작하는가"에 대한 안내가 없다. README를 먼저 읽어야만 동작 순서를 파악할 수 있는데, 이는 확장 특성상 현실적이지 않다.

**권장**:
- MCP가 미연결 상태이고 API 키가 없는 초기 상태에서, Figma 패널에 "시작하기" 안내 텍스트 표시
- 예: "1단계: Connect 버튼을 눌러 MCP 서버에 연결하세요."

### 2-2. MCP 서버 설정 방법이 UI 내에 없음

사용자가 MCP 연결에 실패하면 "Connection failed" 오류가 표시된다. 그러나 MCP 서버를 어떻게 설치하고 실행하는지에 대한 링크나 안내가 UI 어디에도 없다. 설정에서 `figmalab.mcpEndpoint`를 바꾸는 방법도 Panel 내에서는 알 수 없다.

**권장**: 연결 실패 시 "MCP 서버 설정 방법" 링크 또는 "설정 열기" 버튼 노출.

## 3. Agent 패널 UX 문제

### 3-1. Save 버튼이 패널 내부에 없음

현재 구조:
- Save 버튼: 뷰 타이틀 툴바의 아이콘 버튼 (`$(save)`)
- Clear 버튼: 뷰 타이틀 툴바의 아이콘 버튼 (`$(trash)`)

사용자는 API 키와 모델을 입력한 후 어디서 저장하는지 찾지 못할 가능성이 높다. VSCode 뷰 타이틀 툴바 버튼은 마우스를 올렸을 때만 툴팁이 표시되며, 아이콘만으로는 역할이 불명확하다. 패널 내부에 명시적인 "Save" 버튼이 있어야 한다.

```
현재: [타이틀바: 💾 🗑] → 발견하기 어려움
권장: 패널 내부 btn-row에 [Save] [Clear] 버튼 추가
```

### 3-2. API 키를 입력해도 모델을 따로 로드해야 함

사용자 관점의 예상 흐름:
1. API 키 입력 → 자동으로 모델 목록 로드

실제 흐름:
1. API 키 입력 → (아무 반응 없음)
2. 새로고침 버튼 클릭 → 모델 목록 로드

API 키 입력 후 모델을 수동으로 로드해야 하는 추가 단계는 불필요한 마찰이다. API 키 입력 후 포커스가 이동하거나 일정 시간 지연 후 자동으로 모델 목록을 로드하는 것이 더 자연스럽다.

### 3-3. 에이전트 전환 시 사용자에게 의미 없는 안내 메시지

에이전트를 변경하면 다음 notice가 표시된다:
```
"gemini 에이전트로 전환되었습니다. 모델을 다시 로드하세요."
```
"모델을 다시 로드하세요"는 명령이지만 사용자가 해야 할 구체적인 액션을 안내하지 않는다. 새로고침 버튼을 강조하거나 자동으로 모드 전환을 트리거해야 한다.

### 3-4. 상태 텍스트가 중복 정보를 제공

`agent-status` 요소가 표시하는 내용:
```
현재 agent: gemini / model: gemini-1.5-flash
```
이 정보는 이미 위에 있는 select 요소에서 시각적으로 확인 가능하다. 공간 낭비이며 실질적 가치가 낮다.

## 4. Figma 패널 UX 문제

### 4-1. "Source" 패널 제목이 의미 없음

"Source"는 무엇의 source인지 명확하지 않다. "Figma Data Input" 또는 "디자인 데이터" 같은 더 명확한 레이블이 필요하다.

### 4-2. Connect 버튼 로딩 상태 없음

Connect 버튼을 클릭한 후 응답이 올 때까지 (특히 서버가 느리거나 타임아웃이 발생하는 경우) 아무런 시각적 피드백이 없다. 사용자는 버튼을 여러 번 클릭할 수 있다.

**권장**: 클릭 후 버튼을 비활성화하고 "연결 중..." 텍스트 및 로딩 스피너 표시.

### 4-3. Screenshot 버튼 비활성화 이유를 알 수 없음

MCP가 미연결 상태일 때 Screenshot 버튼은 disabled 상태다. 그러나 `title` 속성이나 툴팁이 없어 사용자는 왜 비활성화되어 있는지 알지 못한다.

```typescript
// 현재
if (screenshotBtn) screenshotBtn.disabled = !hasData || !this.connected;

// 개선: disabled 상태에 title 추가
screenshotBtn.title = !this.connected ? 'MCP 연결 후 사용 가능' : '';
```

### 4-4. 데이터 미리보기 영역 크기 제한

```css
/* 현재 */
.code-output {
  max-height: 200px;
}
```

Figma 파일 데이터는 JSON이 수천 줄이 될 수 있다. 200px 제한은 5~6줄만 보여준다. 사이드바 공간 제약 안에서 더 유용하게 사용하려면 최소 높이를 늘리거나, "전체 보기" 링크를 제공해야 한다. (실제로는 에디터에서 열리므로 미리보기는 상징적 역할만 해도 되지만, 이 경우 명확한 "에디터에서 열림" 안내가 필요하다.)

## 5. Prompt 패널 UX 문제

### 5-1. Generate 버튼이 패널 내부에 없음

Agent 패널과 동일한 문제. Generate 버튼은 뷰 타이틀 툴바의 `$(play)` 아이콘에만 있다. 패널을 보는 사용자는 "어디서 실행하는가"를 직관적으로 파악하기 어렵다.

**권장**: Prompt Builder 패널 하단에 명시적인 "Generate" 버튼 추가.

### 5-2. 진행 표시줄이 너무 작고 인위적

```css
/* 현재 */
.progress-track {
  height: 4px;
}
```

4px 높이의 진행 표시줄은 시각적으로 거의 인지되지 않는다. 또한 진행률은 5%씩 고정 증가하여(`progress = Math.min(95, progress + 5)`) 실제 진행 상황을 반영하지 않는다. 스트리밍이 끝날 때까지 95%에서 멈추므로 사용자는 완료가 가까운지 아닌지 알 수 없다.

**권장**:
- 진행 표시줄 높이를 최소 6~8px로 증가
- 실제 완료 불가능한 경우 indeterminate 애니메이션 사용
- 스트리밍 중에는 "생성 중..." 메시지와 경과 시간 표시

### 5-3. 생성 중 취소 기능 없음

코드 생성 중 UI는 완전히 잠기지 않으면서도 취소 버튼이 없다. 긴 생성이 시작되면 사용자는 완료될 때까지 기다리는 수밖에 없다. 비용이 드는 API 호출이므로 취소 기능은 중요하다.

**권장**: 생성 중 "취소" 버튼 표시, AbortController를 통한 스트리밍 중단 지원.

### 5-4. 생성된 코드 영역의 액션 버튼 위치가 어색함

"에디터에서 열기" / "파일로 저장" 버튼이 포함된 `code-actions` div는 두 패널 사이의 독립적인 `btn-row`로 렌더링된다. 이는 "Prompt Builder" 패널과 "Generated Code" 패널 사이에 어색하게 부유하는 UI 요소를 만든다.

```html
<!-- 현재 구조 -->
<div class="panel">Prompt Builder...</div>
<div class="btn-row" id="code-actions">  ← 패널 사이에 버튼이 부유
  <button>에디터에서 열기</button>
  <button>파일로 저장</button>
</div>
<div class="panel">Generated Code...</div>
```

**권장**: 액션 버튼을 "Generated Code" 패널 내부 상단 또는 하단에 배치.

### 5-5. 체크박스 "사용자 프롬프트 포함"의 역할이 모호함

체크박스가 해제되면 textarea가 비활성화되지 않아 사용자는 여전히 텍스트를 입력할 수 있다. 해제되어도 텍스트가 회색 처리되거나 비활성화되지 않으므로 해제 상태의 의미를 인지하기 어렵다.

**권장**: 체크박스 해제 시 textarea를 시각적으로 비활성화(opacity 낮추기 또는 disabled 처리).

## 6. 언어 일관성 문제

현재 UI 전반에 걸쳐 한국어와 영어가 혼재되어 있다. 이는 사용자 경험의 통일성을 해친다.

### 혼재 사례

| 위치 | 현재 텍스트 | 분류 |
|------|------------|------|
| Figma 패널 버튼 | `Connect`, `Fetch Data`, `Screenshot` | 영어 |
| Figma 상태 | `연결됨 (3 tools available)` | 혼용 |
| Agent 패널 제목 | `Agent Setup` | 영어 |
| Agent 상태 | `현재 agent: gemini / model: ...` | 혼용 |
| 모델 셀렉트 기본값 | `-- 모델 로드 --` | 한국어 |
| Prompt 버튼 | `에디터에서 열기`, `파일로 저장` | 한국어 |
| Prompt 패널 제목 | `Prompt Builder`, `Generated Code` | 영어 |
| Notice 메시지 | 대부분 한국어 | 한국어 |

**권장**: UI 언어를 일관되게 한국어 또는 영어 중 하나로 통일. 다국어 지원이 목표라면 i18n 체계를 도입하고, 임시방편으로 혼용하지 않을 것.

## 7. 피드백 및 오류 처리 UX

### 7-1. Notice 메시지가 자동으로 사라지지 않음

성공, 정보 메시지(예: "MCP 데이터를 불러왔습니다.")는 다음 액션 전까지 계속 표시된다. 이로 인해 오래된 상태 메시지가 남아있게 되어 사용자를 혼란스럽게 한다.

**권장**: 성공/정보 notice는 3~5초 후 자동으로 사라지도록 처리.

### 7-2. 오류 메시지가 기술적

```
"MCP fetch failed: connect ECONNREFUSED 127.0.0.1:3845"
```
이런 오류 메시지는 개발자에게는 유용하지만 일반 사용자에게는 의미가 없다. 오류의 원인과 해결 방법을 사용자 언어로 설명해야 한다.

**권장**:
```
"MCP 서버에 연결할 수 없습니다. 서버가 실행 중인지 확인하세요. (포트: 3845)"
```

### 7-3. 코드 생성 오류가 두 곳에 중복 표시

오류 발생 시 `code-output` 영역에 `Error: [message]`가 표시되고, 동시에 `prompt-notice`에도 오류 메시지가 표시된다. 동일한 정보가 두 곳에 나타나 시각적 혼란을 야기한다.

**권장**: 오류는 notice에만 표시하고, code-output 영역은 코드 전용으로 사용.

## 8. 접근성(Accessibility) 문제

### 8-1. 진행 표시줄에 ARIA 지원 없음

```html
<!-- 현재 -->
<div class="progress-track" aria-hidden="true">
  <div class="progress-fill" id="prompt-progress-fill"></div>
</div>
```

`aria-hidden="true"`로 스크린 리더에서 완전히 숨겨졌으나, 생성 진행 상태를 알리는 대체 수단이 없다. `progress-text`는 있지만 live region이 아니므로 상태 변경 시 스크린 리더가 자동으로 읽지 않는다.

**권장**:
```html
<div role="progressbar" aria-valuenow="0" aria-valuemin="0" aria-valuemax="100"
     aria-label="코드 생성 진행률">
```

### 8-2. 상태 도트가 색상에만 의존

```html
<span class="status-dot connected"></span>  <!-- 초록색 -->
<span class="status-dot"></span>           <!-- 빨간색 -->
```

색맹 사용자는 연결 상태를 색상으로만 구분할 수 없다. 텍스트 옆에 도트가 있지만, 도트 자체에는 `aria-label`이 없고 `role`도 없다.

**권장**: `<span class="status-dot" role="img" aria-label="연결됨"></span>`

### 8-3. Notice 영역이 Live Region이 아님

오류나 성공 메시지가 표시될 때 스크린 리더 사용자는 알림을 받지 못한다.

**권장**: notice 요소에 `role="alert"` 또는 `aria-live="polite"` 추가.

### 8-4. 아이콘 전용 버튼에 접근 가능한 레이블 없음

`icon-btn` 클래스 버튼(모델 새로고침 버튼 등)은 `title` 속성만 있다. `title`은 마우스 호버 시에만 표시되며 스크린 리더에서 일관성이 없다.

**권장**: `aria-label` 속성 추가.

## 9. 데이터 지속성 UX 문제

### 9-1. MCP 데이터가 세션 간 유지되지 않음

`lastMcpData`는 메모리에만 저장된다. VSCode를 재시작하면 Figma URL을 다시 입력하고 fetch를 다시 실행해야 한다. 이는 반복 작업 시 상당한 마찰이다.

**권장**: 마지막으로 입력한 Figma URL을 `globalState`에 저장하여 다음 세션에서 자동 복원.

### 9-2. 생성된 코드가 패널 전환 시 손실될 수 있음

`retainContextWhenHidden: true`로 설정되어 있어 실제로는 보존되지만, 사용자는 이를 알 수 없다. 코드를 생성 후 실수로 패널을 닫으면 재생성해야 하는지 불안감을 느낄 수 있다.

**권장**: "생성된 코드는 패널을 닫아도 유지됩니다" 같은 안내 또는 저장 권유 메시지 제공.

### 9-3. 사용자 프롬프트가 저장되지 않음

이전에 작성한 프롬프트가 VSCode 재시작 후 사라진다. 자주 쓰는 지시사항은 `globalState`에 저장하거나, "프롬프트 저장" 기능을 제공하면 유용하다.

## 10. 추가 UX 개선 기회

### 10-1. MCP 엔드포인트 설정을 Figma 패널 내에서 접근 가능하게

현재 `figmalab.mcpEndpoint`는 VSCode 설정(Settings UI)에서만 변경 가능하다. 연결 실패 시 사용자가 이 설정을 변경해야 한다는 것을 UI에서 안내하지 않는다.

**권장**: Figma 패널의 Connect 섹션에 엔드포인트 URL을 표시하고 "설정에서 변경" 링크 추가.

### 10-2. Log 패널의 가시성 부족

오류가 발생해도 Log 패널이 활성화되지 않는다. 오류 발생 시 Log 패널로 자동 이동하거나, 오류 notice에 "로그에서 자세히 보기" 링크를 제공하면 디버깅이 쉬워진다.

### 10-3. 코드 복사 기능 부재

Generated Code 패널에 "복사" 버튼이 없다. "에디터에서 열기"와 "파일로 저장"만 있어서, 빠르게 코드를 클립보드에 복사하려면 수동으로 전체 선택해야 한다.

**권장**: Generated Code 패널에 "클립보드에 복사" 버튼 추가.

### 10-4. Figma URL 유효성 검사 피드백 부재

textarea에 잘못된 URL을 입력해도 즉각적인 피드백이 없다. Fetch Data를 누른 후에야 파싱 결과를 알 수 있다.

**권장**: 입력값이 Figma URL 패턴과 맞지 않을 경우 실시간으로 경고 표시.
